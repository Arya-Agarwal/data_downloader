# 1. Project Overview

This project is a production-grade market data collection system designed specifically for quantitative research in the Indian derivatives market using the Zerodha Kite Connect API.

The primary objective of this repository is **not trading**, **not backtesting**, and **not machine learning**.

Its sole responsibility is to reliably collect, validate, organize, and maintain historical and live market data that can later be used for research.

Think of this repository as the "Data Engineering Layer" of a much larger quantitative research platform.

Every downstream system—including feature engineering, statistical analysis, option chain reconstruction, strategy research, paper trading, and eventually live trading—depends entirely on the quality of the data produced here.

For that reason, correctness, reproducibility, robustness, and maintainability are prioritized over speed or feature count.

---

# 2. Why This Project Exists

Most retail trading projects begin by implementing indicators, signals, or trading strategies directly on broker APIs.

While this approach can produce results quickly, it introduces several long-term problems:

- Missing historical data
- Inconsistent datasets
- Data corruption
- Duplicate records
- Undetected gaps
- Poor reproducibility
- Tight coupling between data collection and strategy logic
- Difficulty validating research results

As research grows in complexity, these issues become increasingly difficult to fix.

This project was created to solve those problems from the beginning.

Instead of building trading logic first, the foundation is a reliable data collection system that ensures every future experiment is built on high-quality, validated market data.

# 6. Core Design Principles

Every design decision in this repository follows the same philosophy.

## Single Responsibility

Each module performs exactly one job.

Examples:

- Authentication handles authentication.
- Tracker maintains metadata.
- Validator validates datasets.
- Gap Recovery repairs datasets.
- WebSocket receives ticks.
- Tick Writer stores ticks.

Responsibilities never overlap.

---

## Data First

Research quality is determined by data quality.

Whenever a trade-off exists between convenience and data integrity, data integrity always wins.

---

## Production Before Research

A reliable downloader is more valuable than a sophisticated trading strategy built on unreliable data.

This repository focuses on building infrastructure before analytics.

---

## Fail Safely

Unexpected failures should never silently corrupt datasets.

Potential failures are logged, validated, and recoverable wherever possible.

---

## Deterministic Behaviour

Running the downloader multiple times on the same inputs should produce identical outputs.

No module should behave differently simply because it is executed again.

---

## Loose Coupling

Modules communicate through clearly defined interfaces.

Changing one module should require minimal or no changes elsewhere.

This makes the project easier to extend over time.

# 13. Startup Sequence

Immediately after the program starts, several mandatory initialization steps are executed.

## Step 1 — Authentication

The bot authenticates using the Zerodha Kite Connect API.

Responsibilities:

- Validate stored access token
- Generate Kite client
- Create authenticated KiteTicker
- Verify login

No market data is downloaded before authentication succeeds.

If authentication fails, execution stops immediately.

---

## Step 2 — Instrument Master

The complete Zerodha instrument dump is loaded.

If today's instrument master already exists:

```
Reuse cached file
```

Otherwise:

```
Download new instrument master
```

This file contains:

- Instrument Token
- Trading Symbol
- Exchange
- Segment
- Expiry
- Strike
- Lot Size

Every downstream module depends on this file.

---

## Step 3 — Tracker Initialization

The tracker maintains metadata for every downloaded dataset.

If it does not exist:

```
tracker.parquet
```

is created automatically.

The tracker stores:

- Download status
- Validation status
- Recovery status
- Last timestamps
- File locations
- Row counts

The tracker never stores market data.

It stores metadata about market data.

---

## Step 4 — Expired Contract Update

Expired contracts inside the tracker are marked automatically.

No historical files are deleted.

Only tracker metadata changes.

---

## Step 5 — Startup Validation

Several sanity checks are performed before the downloader begins.

Typical checks include:

- Required folders exist
- Instrument master exists
- Tracker exists
- Configuration is valid
- Runtime environment is healthy

Any critical failure stops execution immediately.

The downloader follows several architectural principles.

Each module should perform one task only.

## Recover Later

Historical gaps can usually be repaired later.

Live ticks cannot.

Therefore:

```
Live collection always has higher priority.
```

---

## Atomic Writes

Files should never be left partially written.

Every parquet write should either:

Complete successfully

or

Leave the previous version intact.

---

## Idempotency

Running the downloader multiple times should not duplicate historical data.

Missing data is downloaded.

Existing data is preserved.

# 58. Current Project Scope

The downloader is responsible only for collecting market data.

Included:

- Authentication
- Instrument master
- Historical downloads
- Incremental updates
- Gap recovery
- Validation
- Live WebSocket collection
- Tick storage
- Dynamic option subscriptions
- Runtime monitoring
- Graceful shutdown
- End-of-day reconciliation

Not included:

- Strategy generation
- Signal engine
- Indicators
- PCA
- Machine learning
- Backtesting
- Trade execution
- Order placement
- Portfolio management
- Research features

Those belong in a separate project that consumes the data produced by this downloader.

# 59. Future Extension Points

The architecture has been designed so that future functionality can be added without changing the downloader's core responsibilities.

Examples include:

- Candle generation from live ticks
- Data quality dashboards
- Automated integrity reports
- Compression and archival
- Database ingestion
- Research pipelines

These should consume the downloader's outputs rather than modify its internals.
